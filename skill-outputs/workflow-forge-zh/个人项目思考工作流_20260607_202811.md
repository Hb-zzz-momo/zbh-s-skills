# 个人项目思考工作流

> 一句话说明：把项目从“我想做/我要交付”推进到“边界清楚、能跑、可验证、能交接”的固定工作流。

## 前置条件

- 有一个项目目标、题面、需求、代码仓库、实验材料或用户口头目标。
- 明确当前任务属于哪类：工程开发、数学建模、科研实验、课程实验、论文写作、软著、PPT、资料调研或长任务运行。
- 默认不相信 AI 的自由发挥，要求先绑定真实材料和当前文件树。

## 总流程

```text
Define 定义边界
  -> Plan 拆执行路径
  -> Build 做最小可运行产物
  -> Verify 用证据验收
  -> Review 审风险和结论边界
  -> Ship 交付、归档、接班
```

## Step 1: Define 定义边界

- **做什么**：先确认任务到底要解决什么、不能做什么、最终交付什么。
- **输入**：用户需求、题面、prompt 文件、计划文档、已有代码、数据、模板、历史约束。
- **输出**：目标、范围、不做事项、验收标准、关键风险。
- **完成标准**：能用 5 句话讲清楚“要做什么、依据什么、输出到哪里、怎么验收、哪些不能擅自扩展”。
- **注意事项**：这一步最容易出错的是 AI 自行补假设，比如扩大参数范围、改输出格式、改文件结构、替用户决定论文结论。

### Define 检查点

- [ ] 是否已经读了用户给的真实材料，而不是凭经验回答。
- [ ] 是否明确输出目录、文件名、运行命令或报告格式。
- [ ] 是否列出“不做什么”。
- [ ] 是否存在缺失、冲突或不确定点。
- [ ] 如果有固定计划，是否先锁定计划边界。p

## Step 2: Plan 拆执行路径

- **做什么**：把目标拆成可执行、可检查的小阶段。
- **输入**：Define 阶段的目标、范围、验收标准。
- **输出**：分阶段任务表、依赖关系、每阶段检查点。
- **完成标准**：每个步骤都有明确输入、输出和验收方式；不能出现“一步完成全部系统”。
- **注意事项**：计划要服务执行，不能变成空泛路线图。

### Plan 推荐拆法

```text
1. 读材料 / 扫目录 / 找入口
2. 确认现有链路或数据结构
3. 做最小可运行基线
4. 增量补功能或实验
5. 跑 smoke 验证
6. 跑 formal 或完整验证
7. 输出报告 / 表格 / 图 / 文档
8. 自检边界和风险
```

### Plan 检查点

- [ ] 是否先做 smoke，而不是直接跑耗时 formal。
- [ ] 是否每一步都能单独验证。
- [ ] 是否把“读现有代码/材料”放在“修改/生成”之前。
- [ ] 是否为长任务设计日志、进度文件或中间检查点。
- [ ] 是否把论文/报告结论放到结果审计之后。

## Step 3: Build 做最小可运行产物

- **做什么**：按小步实现，不追求一开始就完美，先让主链路闭合。
- **输入**：计划步骤、当前代码、数据、模板、已有约定。
- **输出**：可运行脚本、代码改动、结果文件、报告草稿、图表或项目结构。
- **完成标准**：至少有一个真实命令能跑通，并产生可检查产物。
- **注意事项**：不要顺手重构无关代码，不要移动用户没要求移动的文件。

### Build 执行习惯

- 先读目录和入口文件。
- 优先沿用现有框架、命名、数据结构和输出习惯。
- 先做基线，再调效果。
- 先写脚本和产物，再解释。
- 对中文报告、建模任务、科研任务，优先生成可复用文件而不是只在聊天里讲。

### Build 检查点

- [ ] 代码是否写在用户指定目录。
- [ ] 是否保留了原有文件树约束。
- [ ] 是否有可复制运行命令。
- [ ] 是否生成了真实输出文件。
- [ ] 是否避免了无关重构。

## Step 4: Verify 用证据验收

- **做什么**：用测试、运行日志、文件检查、抽样核对或浏览器验证证明结果是真的。
- **输入**：Build 阶段产物。
- **输出**：测试结果、运行日志、结果摘要、错误清单、验收结论。
- **完成标准**：能回答“它怎么证明自己是对的”。
- **注意事项**：不能只说“应该可以”；必须给命令、输出文件、样本核查或截图/日志依据。

### Verify 常用方式

- 工程代码：`python -m pytest`、构建命令、lint、接口请求。
- 前端页面：启动 dev server，用浏览器验证控制台、网络、交互、布局。
- 建模/科研：smoke 先跑小样本，formal 再跑完整配置。
- 数据结果：检查 raw、summary、manifest、字段含义、行数、缺失值、指标方向。
- 文档报告：核对是否与实际代码和结果一致。

### Verify 检查点

- [ ] 是否跑了最小验证。
- [ ] 是否说明验证命令和结果。
- [ ] 是否检查输出文件存在且内容合理。
- [ ] 是否区分 smoke 结果和 formal 结果。
- [ ] 是否记录无法验证的原因。

## Step 5: Review 审风险和结论边界

- **做什么**：在交付前审查 bug、回归、安全、性能、论文 claim、计划偏离和可维护性。
- **输入**：代码 diff、运行结果、报告草稿、实验结果、用户原始边界。
- **输出**：风险清单、修正建议、是否可交付判断。
- **完成标准**：明确指出是否存在阻塞项；没有问题也要说明剩余风险。
- **注意事项**：Review 不是夸成果，而是找可能出事的地方。

### Review 重点问题

- 是否偏离用户给的计划、题面或输出格式。
- 是否把诊断结果写成了正式结论。
- 是否只有 summary 没有 raw 支撑。
- 是否存在数据泄漏、权限绕过、公式注入、密钥暴露。
- 是否为了性能或指标堆了不可解释机制。
- 是否缺少关键测试。
- 是否修改了不相关文件。

### Review 检查点

- [ ] 是否有边界偏离审计。
- [ ] 是否有结果证据链。
- [ ] 是否有代码质量和测试缺口检查。
- [ ] 是否对论文/报告 claim 做了降级或限定。
- [ ] 是否判断需要停止迭代、继续优化或冻结当前成果。

## Step 6: Ship 交付、归档、接班

- **做什么**：把结果整理成用户能继续使用的状态。
- **输入**：最终代码、结果文件、验证记录、风险结论。
- **输出**：交付说明、运行命令、输出路径、报告文件、接班文档或归档索引。
- **完成标准**：用户不需要追问“文件在哪、怎么跑、结果可信不可信、下一步做什么”。
- **注意事项**：交付不是把话说完，而是把后续使用链路闭合。

### Ship 输出模板

```text
已完成：
- 改了什么
- 生成了什么
- 文件在哪里

验证：
- 跑了什么命令
- 结果是什么

使用方式：
- 工作目录
- 运行命令
- 输出位置

风险/下一步：
- 未验证项
- 需要人工确认项
- 后续优先级
```

## 异常处理

| 异常场景 | 处理方式 |
|---|---|
| 材料缺失或互相冲突 | 停止补假设，列出具体问题让用户确认 |
| 用户给了固定计划 | 使用边界锁，不扩展参数、文件结构、结论口径 |
| 长任务耗时 | 先 smoke，再后台 formal；保留日志、进度和 checkpoint |
| 结果不好 | 先审指标、raw、配置和实现，不急着堆新机制 |
| 连续多轮无改进 | 停止小修小补，做停滞复盘 |
| 报告要写结论 | 先审证据链，再写 paper-safe 表述 |
| 文件编码乱码 | 先确认源编码，再修复源文件并重新生成索引 |
| 用户只问解释 | 以代码/文件/字段链路解释，不泛泛讲理论 |

## 产出物清单

- [ ] 目标和边界说明
- [ ] 分阶段计划
- [ ] 可运行代码或可检查产物
- [ ] 验证命令和验证结果
- [ ] 风险审查结论
- [ ] 交付说明和输出路径
- [ ] 必要时的接班文档或归档索引

## 可直接复制给 AI 的提示词

```text
请按我的项目工作流执行：
1. Define：先读当前材料和文件树，锁定目标、边界、输出和验收标准。
2. Plan：拆成可验证的小阶段，先 smoke 后 formal，不要一次性大改。
3. Build：先做最小可运行链路，沿用现有结构，只改必要文件。
4. Verify：给出实际运行命令、输出路径和检查结果，不能只说“应该可以”。
5. Review：审查是否偏离计划、是否有测试缺口、结果是否支撑结论。
6. Ship：最后给我文件位置、运行方式、验证结论、风险和下一步。
如果发现缺失、冲突或需要新增假设，先问我，不要自行补。
```

## 和 Define -> Plan -> Build -> Verify -> Review -> Ship 的对应关系

| 通用阶段 | 你的项目化表达 | 关键问题 |
|---|---|---|
| Define | 先读材料，锁目标和边界 | 我到底要交付什么，不能擅自做什么？ |
| Plan | 拆成 smoke/formal/报告等阶段 | 怎么一步步做，每步怎么验收？ |
| Build | 先做最小可运行链路 | 有没有真实代码、脚本或产物？ |
| Verify | 用命令、日志、文件、抽样证明 | 结果怎么证明是真的？ |
| Review | 审计划偏离、风险和 claim | 有没有越界、过拟合、缺测试或夸大结论？ |
| Ship | 给路径、命令、说明和接班 | 用户能不能接着跑、接着写、接着交付？ |

## 最短记忆版

```text
先读真材料 -> 锁边界 -> 拆小步 -> 做基线 -> 跑验证 -> 审风险 -> 给路径和命令 -> 沉淀经验
```
