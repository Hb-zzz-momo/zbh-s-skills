# 本 skills 项目工作流小白版

> 一句话说明：这个 skills 项目不是一堆提示词，而是一套“先路由、再执行、再验证、再沉淀”的本地 AI 协作系统。

## 来源和优化说明

本文件是在已有 `skill-outputs/` 材料基础上优化整合而来：

- `skill-outputs/project-dev-zh/skills架构全量分析_20260607_201308.md`
- `skill-outputs/workflow-forge-zh/个人项目思考工作流_20260607_202811.md`
- `skill-outputs/workflow-forge-zh/科研Skill调用流整合方案_v3_20260608_120614.md`
- 最新本地 skill：`research-skill-flow-zh`、`engineering-skill-flow-zh`、`subagent-orchestration-zh`

优化重点：

- 把“架构分析”改成“能直接用的工作流”。
- 把科研流、工程流、子代理流和 skill 维护流统一到同一种小白解释方式。
- 每条工作流都给一个生活化/项目化例子，说明为什么要这么走。

## 总览：本项目最重要的 7 条工作流

```text
1. 全库路由工作流：先判断该用哪个 skill
2. 通用项目执行工作流：Define -> Plan -> Build -> Verify -> Review -> Ship
3. 工程多方案实现工作流：多方案 -> Gate -> spike -> 主路径实现
4. 科研多候选工作流：多痛点 -> Gap Gate -> smoke -> formal -> claim
5. 子代理编排工作流：父代理控主线，子代理只做并行支线
6. Skill 维护工作流：改 skill -> 接路由 -> 生成 README -> 校验
7. 产物归档和记忆工作流：输出文件、验证证据、下次继承约束
```

最短记忆版：

```text
先选路 -> 锁边界 -> 多候选 -> 小验证 -> 正式做 -> 审风险 -> 给路径和命令 -> 沉淀经验
```

## 工作流 1：全库路由工作流

### 它解决什么问题

防止 AI 一上来就乱用 skill，或者一次性加载一大堆无关 skill。

### 流程

```text
用户请求
-> 判断领域
-> 判断阶段
-> 判断风险
-> 选择 1 个主 skill + 0-2 个辅助 skill
-> 执行
```

### 小白例子

你说：

```text
帮我给项目加一个订单导出功能
```

错误做法：

```text
直接开始写代码。
```

正确做法：

```text
先判断：
- 这是工程开发，不是科研、论文、PPT。
- 它可能涉及前端按钮、后端接口、Excel 文件、权限。
- 如果需求复杂，先走 engineering-skill-flow-zh。
- 如果只是小改，走 project-dev-zh + incremental-implementation。
```

### 检查点

- 是否已经判断任务属于哪个领域。
- 是否只选了必要 skill。
- 是否没有把科研审计、论文写作、工程 review 混用。
- 是否有固定计划需要先触发 `plan-faithful-execution-zh`。

## 工作流 2：通用项目执行工作流

### 它解决什么问题

防止项目从“想做”直接跳到“瞎写”，最后用户还要追问：文件在哪、怎么跑、结果可信吗。

### 流程

```text
Define
-> Plan
-> Build
-> Verify
-> Review
-> Ship
```

### 小白例子

需求：

```text
做一个学生成绩分析脚本。
```

#### Define：先问清楚要什么

```text
要读哪个 Excel？
输出 CSV 还是图？
分析平均分、排名，还是挂科率？
结果保存到哪里？
```

#### Plan：拆成小步骤

```text
1. 读取 Excel
2. 检查字段
3. 计算平均分和排名
4. 生成结果表
5. 跑一次样例验证
```

#### Build：先做最小可运行版本

```text
先写一个能读取 Excel 并输出平均分的脚本。
不要一开始就做图表、网页、数据库。
```

#### Verify：用证据证明能跑

```text
运行 python analyze_scores.py
检查 outputs/scores_summary.csv 是否生成。
抽 3 个学生手算核对。
```

#### Review：审风险

```text
有没有空分数？
有没有把字符串当数字？
是否改了无关文件？
```

#### Ship：交付

```text
告诉用户：
- 脚本在哪
- 怎么运行
- 输出在哪
- 哪些情况没覆盖
```

### 检查点

- 是否先读材料和文件树。
- 是否有真实命令。
- 是否有真实输出路径。
- 是否说明风险和下一步。

## 工作流 3：工程多方案实现工作流

### 它解决什么问题

防止复杂工程任务变成“一边重构、一边加功能、一边优化性能”，最后不知道哪个改动真的有效。

### 流程

```text
Project Intake
-> Solution Gate
-> Implementation Design
-> Candidate Spike / Slice Funnel
-> Main Build
-> Verify / Review
-> Ship / Memory
```

### 小白例子

需求：

```text
给后台管理系统增加订单导出 Excel。
```

#### Project Intake：先看项目现状

```text
看订单接口在哪。
看前端有没有类似下载按钮。
看后端有没有 Excel 依赖。
看权限系统怎么做。
```

#### Solution Gate：比较多个方案

| 方案 | 做法 | 判断 |
|---|---|---|
| A | 前端直接导出当前表格 | BLOCK，只能导出当前页 |
| B | 后端生成 Excel，前端下载 | PASS，满足完整导出 |
| C | 后台异步生成文件 | CAUTION，适合大数据但本轮太重 |

#### Implementation Design：把选中方案写清楚

```text
GET /api/orders/export
参数：startDate, endDate
权限：管理员
返回：Excel 文件
```

#### Candidate Spike / Slice Funnel：先做小实验

```text
先验证后端能不能生成一个 10 行 Excel。
再验证浏览器能不能下载。
```

#### Main Build：正式实现

```text
切片 1：后端导出接口
切片 2：前端导出按钮
切片 3：权限和错误处理
```

#### Verify / Review：验收和挑问题

```text
管理员能下载。
普通用户不能下载。
Excel 字段正确。
日期参数错误时有提示。
```

#### Ship / Memory：交付和记忆

```text
已验证：普通订单导出。
未覆盖：10 万条以上异步导出。
下次约束：大数据量不要继续用同步导出。
```

### 检查点

- 是否允许早期多个方案。
- 是否进入正式开发前收敛到一个主方案。
- 是否每个 spike 只验证一个候选。
- 是否没有把未验证候选写成已完成成果。

## 工作流 4：科研多候选工作流

### 它解决什么问题

防止科研任务从“想到一个创新点”直接跳到“改代码跑实验”，最后发现痛点不存在、贡献不清楚、结果不能写论文。

### 流程

```text
Memory Intake
-> Gap Gate
-> Research Design
-> Candidate Smoke Funnel
-> Dev / Formal
-> Paper Memory / Writing
-> Handoff / Archive
```

### 小白例子

任务：

```text
改进一个特征选择算法，想发论文。
```

#### Memory Intake：先读历史和文献

```text
看以前试过哪些方法。
看哪些方法失败了。
看 raw、summary、日志。
看相关论文是否已经做过类似创新点。
```

#### Gap Gate：判断 gap 真的存在吗

| 候选 | 痛点 | 创新点 | 判断 |
|---|---|---|---|
| C1 | 收敛慢 | 加自适应搜索步长 | PASS |
| C2 | 特征冗余 | 加相关性惩罚 | CAUTION，需要看消融 |
| C3 | 指标不好 | 堆一个新模块 | BLOCK，机制说不清 |

#### Research Design：定义怎么验证

```text
C1 的失败条件：
- smoke 后没有提升
- runtime 明显膨胀
- 消融不能说明机制有效
```

#### Candidate Smoke Funnel：每个候选单独 smoke

```text
只改 C1 跑 smoke。
只改 C2 跑 smoke。
不要 C1+C2+C3 一起改完再说整体提升。
```

#### Dev / Formal：只让主候选进入正式实验

```text
如果 C1 smoke 最稳，就 C1 进 dev/formal。
C2 留作后续或次要消融。
```

#### Paper Memory / Writing：论文 claim 分级

```text
主创新点：formal 支持，可以写贡献。
次创新点：只有 dev 支持，只能谨慎写。
失败候选：写局限或未来工作。
```

### 检查点

- 是否保留多个痛点和多个假设。
- 是否每个创新点有独立证据和失败条件。
- 是否没有把多个机制混改后直接 claim。
- formal 或论文主表是否触发结果审计。

## 工作流 5：子代理编排工作流

### 它解决什么问题

防止“多代理”变成一堆代理同时乱改文件，最后父代理也不知道谁改了什么。

### 流程

```text
确认授权
-> 选本地主 skill
-> 拆父子职责
-> 写 worker brief
-> 并行执行
-> 父代理回收
-> 本地审计验收
```

### 小白例子

你说：

```text
/sub 帮我并行检查这个项目的测试缺口和安全风险。
```

正确拆法：

```text
父代理：
- 控制主流程
- 决定最终结论
- 不让子代理越界改代码

explorer 1：
- 只读扫描测试缺口

explorer 2：
- 只读扫描安全风险

late reviewer：
- 最后只读 review 父代理整理的结论
```

错误拆法：

```text
worker A 改登录模块。
worker B 也改登录模块。
worker C 顺手重构配置。
```

因为这样会写范围冲突。

### 检查点

- 用户是否明确授权 subagent。
- 子代理是否默认只读。
- worker 是否有互斥写范围。
- 子代理结论是否回到父代理验收。
- 科研 claim 是否仍回到本地审计 skill。

## 工作流 6：Skill 维护工作流

### 它解决什么问题

防止新增或修改 skill 后，README 不更新、路由搜不到、frontmatter 不合法、技能职责和旧 skill 冲突。

### 流程

```text
读现有 skill 和路由
-> 判断是新增还是修改
-> 写 SKILL.md
-> 接入 skill-router / SKILL_ROUTING / README 生成器
-> 重新生成 README
-> quick_validate 校验
-> git diff 检查
```

### 小白例子

任务：

```text
新增一个工程调用流 skill。
```

正确做法：

```text
1. 先看已有 project-dev-zh、agent-skills-ao、research-skill-flow-zh。
2. 确认不是重复造一个 project-dev-zh。
3. 新建 engineering-skill-flow-zh。
4. 在 skill-router-zh 里加触发条件。
5. 在 SKILL_ROUTING.zh-CN.md 里加领域和组合。
6. 在 Update-SkillsReadme.ps1 加 scenarioBySkill 映射。
7. 重新生成 README。
8. quick_validate 检查新 skill 和 router。
```

错误做法：

```text
只新建一个 SKILL.md，不接路由，不生成 README。
```

这样后续 AI 很可能找不到它。

### 检查点

- `name` 是否和目录名一致。
- `description` 是否能触发正确场景。
- 是否和现有 skill 职责冲突。
- 是否重新生成 `skills/README.md`。
- 是否通过 `quick_validate.py`。
- 是否没有手改生成物后忘记改生成器。

## 工作流 7：产物归档和记忆工作流

### 它解决什么问题

防止每次任务结束后只在聊天里说“完成了”，下次换窗口或换 AI 就不知道文件在哪、命令是什么、哪些坑不能再踩。

### 流程

```text
任务产物
-> 写入 skill-outputs/<skill-name>/
-> 记录命令和结果
-> 标注风险和未覆盖项
-> 必要时生成 handoff
-> 下次从状态卡继承
```

### 小白例子

任务：

```text
整理一次科研实验结果。
```

不要只说：

```text
结果整理好了。
```

要留下：

```text
skill-outputs/research-experiment-ops-zh/实验结果整理_20260608_183000.md

里面写清楚：
- 尝试方法
- 计划
- 脚本/命令
- 结果
- 经验
- 下次必须继承的约束
- raw/summary/config 路径
- 哪些结果不能写成论文 claim
```

工程任务同理：

```text
已验证功能：
- 订单导出接口

已实现未充分验证：
- 大数据量导出性能

失败候选：
- 前端直接导出完整订单

下次约束：
- 超大导出要走异步任务
```

### 检查点

- 是否给出文件路径。
- 是否给出运行命令。
- 是否记录验证结果。
- 是否记录失败路线。
- 是否写清下次必须继承的约束。

## 这 7 条工作流之间怎么配合

### 普通工程任务

```text
全库路由
-> 通用项目执行
-> 如果复杂，进入工程多方案实现
-> Verify / Review
-> 产物归档和记忆
```

例子：

```text
给项目加导出功能
-> 先路由到工程
-> 复杂则用 engineering-skill-flow-zh
-> 小实验验证 Excel 下载
-> 正式实现
-> 测试和 review
-> 写清文件、命令、风险
```

### 科研任务

```text
全库路由
-> 科研多候选工作流
-> 实验执行和审计
-> 论文 claim 分级
-> 产物归档和记忆
```

例子：

```text
想改算法发论文
-> 先读历史实验和文献
-> 列多个痛点和创新点
-> Gap Gate 筛候选
-> 每个候选单独 smoke
-> 只让主候选 formal
-> 结果审计后再写论文贡献
```

### 明确要求 subagent 的任务

```text
全库路由
-> 子代理编排
-> 父代理保留主 skill
-> 子代理只做可并行支线
-> 父代理回收和审计
```

例子：

```text
/sub 并行审这个项目
-> explorer A 看代码路径
-> explorer B 看测试缺口
-> explorer C 看安全风险
-> 父代理整合，不让子代理直接决定最终结论
```

### 修改 skills 库本身

```text
全库路由
-> Skill 维护工作流
-> 改 SKILL.md
-> 接 router 和 README 生成器
-> quick_validate
-> 产物归档和记忆
```

例子：

```text
新增工程调用流
-> 新建 engineering-skill-flow-zh
-> 更新 skill-router-zh
-> 更新 SKILL_ROUTING.zh-CN.md
-> 更新 Update-SkillsReadme.ps1
-> 生成 README
-> 验证 skill 合法
```

## 最容易踩的坑

| 坑 | 表现 | 正确处理 |
|---|---|---|
| 不路由直接开做 | AI 上来就写代码或写论文 | 先用 `skill-router-zh` 判断领域和阶段 |
| 多方案混成一个实现 | 重构、功能、性能一起改 | 复杂工程先用 `engineering-skill-flow-zh` |
| 科研直接 claim | smoke 一好就写贡献 | formal 和审计后再分级 claim |
| subagent 乱并行 | 多个 worker 改同一文件 | 写范围冲突时退化为串行 |
| 改 skill 不更新 README | README 找不到新 skill | 改 `Update-SkillsReadme.ps1` 后重新生成 |
| 只聊天不落文件 | 下次找不到结果 | 写入 `skill-outputs/<skill-name>/` |
| 忘记继承失败经验 | 换名字重复踩坑 | 六段式记忆必须保留 |

## 一句话总规则

```text
本 skills 项目的工作流不是为了让 AI 多想，而是为了让 AI 少乱想：
先选对路，再列候选，再小验证，再正式做，最后用证据和文件把结果交清楚。
```

## 可直接复制给 AI 的提示词

```text
请按本 skills 项目工作流执行：
1. 先用全库路由判断领域、阶段和风险，只选必要 skill。
2. 如果是复杂工程，先走 engineering-skill-flow-zh，列多个实现候选，通过 Gate 后再 spike。
3. 如果是科研任务，先走 research-skill-flow-zh，读历史和文献，列多个痛点、假设和创新点，再 smoke/formal。
4. 如果我明确要求 subagent，先走 subagent-orchestration-zh，父代理保留最终集成和验收。
5. 如果修改 skills 库本身，必须更新路由、README 生成器，重新生成 README，并跑 quick_validate。
6. 任务结束时，把路径、命令、验证结果、风险和下次必须继承的约束写清楚。
不要把未验证候选写成正式成果；发现缺口或冲突时先问我。
```
