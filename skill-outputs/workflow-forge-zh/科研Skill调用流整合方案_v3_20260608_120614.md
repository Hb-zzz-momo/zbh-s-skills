# 科研 Skill 调用流整合方案 v3

> 目标：把科研任务从“想到一个点就直接做”改成“多候选生成、分阶段筛选、单实验验证、证据审计后再写论文”的可执行调用流。

## 1. 总原则

本流程以本地科研 skill 为主干，外部科研 skill 只作为精选辅助或模板参考。

每次科研任务的调用预算固定为：

```text
1 个主 skill + 最多 2 个辅助 skill + 条件触发的审计 skill
```

审计 skill 不计入辅助 skill 数量，但必须满足触发条件才调用，不能常驻加载。

## 2. 角色边界

### 2.1 主 skill

主 skill 是当前阶段的责任主体。一次任务只能有一个主 skill。

| 阶段 | 主 skill |
|---|---|
| Memory Intake / Gap Gate / Research Design | `research-coach-zh` |
| 实验计划与执行 | `research-experiment-ops-zh` |
| 论文写作 | `paper-writing-zh` |
| 项目交接 | `project-handoff-zh` |
| 产物整理 | `artifact-curator-zh` |

### 2.2 辅助 skill

辅助 skill 只提供材料、模板、证据或执行支持，不能改变主流程规则。

常见辅助：

```text
info-evidence-chain-zh
literature-triage-matrix
research-design-helper
paper-memory-builder
research-statistics-reporting-zh
agent-wait-monitor-zh
artifact-curator-zh
```

### 2.3 审计 skill

审计 skill 条件触发，不算辅助 skill。

| 审计 skill | 触发条件 |
|---|---|
| `decisive-result-audit-zh` | formal/gate、冻结版本、论文主表、算法胜负、结果能否支撑 claim |
| `research-iteration-audit-zh` | smoke/dev 结果后、机制堆叠、连续失败、创新点说不清、论文前创新点体检 |
| `iteration-reflection-guard-zh` | 提新方案、继续迭代、需要吸取历史经验、连续无效优化 |
| `plan-faithful-execution-zh` | 用户给了计划、题面、prompt、参数边界、固定输出格式 |

## 3. 完整科研调用流

```text
Route
  -> Memory Intake
  -> Gap Gate
  -> Research Design
  -> Candidate Smoke Funnel
  -> Dev / Formal
  -> Result Audit
  -> Iteration Decision
  -> Paper Memory / Writing
  -> Handoff / Archive
```

## 4. Step 1：Route

- **做什么**：判断本轮科研任务处于哪个阶段，只选择必要 skill。
- **输入**：用户请求、当前项目状态、是否有实验结果、是否准备写论文。
- **输出**：1 个主 skill、最多 2 个辅助 skill、可能触发的审计 skill。
- **完成标准**：能写出本轮 skill 组合和选择理由。

### Route 检查点

- [ ] 是否只有 1 个主 skill。
- [ ] 辅助 skill 是否不超过 2 个。
- [ ] 审计 skill 是否有明确触发条件。
- [ ] 外部 skill 是否只是辅助或模板参考。

## 5. Step 2：Memory Intake

主 skill：

```text
research-coach-zh
```

最多辅助：

```text
info-evidence-chain-zh
literature-triage-matrix
```

### 做什么

进入任何科研设计或实验前，先读取历史和证据，不允许直接提出方案。

必须读取：

- 历史实验记录。
- 实验日志、summary、raw。
- 既有失败路线和有效路线。
- 文献材料、文献矩阵、证据链材料。
- 已有 `skill-outputs/`、Tracing、项目计划文档。

### 输出 1：项目状态卡

项目状态卡必须保留六段式结构：

```markdown
## 尝试方法
- ...

## 计划
- ...

## 脚本/命令
- ...

## 结果
- ...

## 经验
- ...

## 下次必须继承的约束
- ...
```

可追加字段：

```markdown
## 证据链路径
- raw:
- summary:
- metric:
- report:

## 失败边界
- ...

## claim 允许范围
- 允许：
- 禁止：

## stale 结果标记
- ...
```

### 输出 2：候选记录区

```markdown
## 候选痛点列表
| ID | 痛点 | 证据 | 影响指标 | 备注 |
|---|---|---|---|---|

## 候选假设列表
| ID | 对应痛点 | 假设 | 可验证预测 | 风险 |
|---|---|---|---|---|

## 候选创新点列表
| ID | 对应假设 | 创新点 | 动机 | 预期收益 | 实现成本 | 可消融方式 | 失败条件 |
|---|---|---|---|---|---|---|---|
```

### Memory Intake 检查点

- [ ] 是否读取历史实验记录。
- [ ] 是否读取实验日志、summary、raw。
- [ ] 是否读取文献或文献矩阵。
- [ ] 是否生成项目状态卡。
- [ ] 是否保留六段式记忆。
- [ ] 是否列出候选痛点、候选假设、候选创新点。

## 6. Step 3：Gap Gate

主 skill：

```text
research-coach-zh
```

最多辅助：

```text
literature-triage-matrix
info-evidence-chain-zh
```

### 做什么

对多个痛点、多个假设、多个创新点做候选筛选，而不是一开始收敛到单点。

### 多候选 Gate

| Gate | 判断问题 | 主要依据 |
|---|---|---|
| Gate 1 | 每个痛点对应的 gap 是否存在？ | 文献矩阵参考 |
| Gate 2 | 每个创新点是否构成真实贡献？ | 文献对比辅助 |
| Gate 3 | 每个候选以当前数据/算力/时间是否可行？ | 本地实验经验为主 |

### 输出：候选池表

```markdown
| 候选ID | 痛点 | 假设 | 创新点 | 文献支持 | 本地可行性 | 预期收益 | 实现成本 | 风险等级 | 建议动作 |
|---|---|---|---|---|---|---|---|---|---|
| C1 | ... | ... | ... | ... | ... | ... | ... | 高/中/低 | PASS / CAUTION / BLOCK |
```

### Gate 判定

```text
PASS：可以进入 Research Design
CAUTION：只能预研、只读分析或低成本 smoke
BLOCK：不进入实验，不进入论文 claim
```

### Gap Gate 检查点

- [ ] 每个候选是否有独立 ID。
- [ ] 是否使用文献矩阵判断 gap。
- [ ] 是否用本地实验经验判断可行性。
- [ ] 是否给出 PASS / CAUTION / BLOCK。
- [ ] 是否禁止 BLOCK 候选进入实验。

## 7. Step 4：Research Design

主 skill：

```text
research-coach-zh
```

最多辅助：

```text
research-design-helper
literature-triage-matrix
```

### 做什么

把通过 Gap Gate 的候选转成可实验、可消融、可审计的设计。

外部 `research-design-helper` 只提供模板参考：

```text
研究问题
候选痛点
候选假设
候选机制
可识别性
验证计划
风险登记
```

### 输出：创新点可行性矩阵

```markdown
| 创新点 | 对应痛点 | 预期机制 | 可独立实现 | 可独立消融 | smoke 成本 | 失败条件 |
|---|---|---|---|---|---|---|
| ... | ... | ... | 是/否 | 是/否 | 低/中/高 | ... |
```

### 筛选规则

- 初始可以保留多个候选。
- 进入 smoke 前必须给每个候选定义最小验证。
- dev/formal 前必须收敛到一个主候选或一个明确组合候选。
- 组合候选必须说明为什么不能拆开验证。
- 组合候选必须有组合 ID，例如 `C1+C3`，不能写成未记录混合机制。

### Research Design 检查点

- [ ] 是否保留多个候选用于筛选。
- [ ] 每个候选是否有最小验证方式。
- [ ] 每个创新点是否可独立实现或解释不可拆原因。
- [ ] 每个创新点是否可独立消融或解释不可拆原因。
- [ ] 是否定义失败条件。

## 8. Step 5：Candidate Smoke Funnel

主 skill：

```text
research-experiment-ops-zh
```

最多辅助：

```text
agent-wait-monitor-zh
artifact-curator-zh
```

审计 skill 条件触发：

```text
plan-faithful-execution-zh
iteration-reflection-guard-zh
research-iteration-audit-zh
```

### 做什么

用 smoke 漏斗筛选多个候选。

```text
候选池
  -> 每个候选独立 smoke
  -> smoke 结果排序
  -> 选择 1 个主候选进入 dev
  -> dev 通过后进入 formal
  -> formal 后审计 claim
```

### 本地硬规则

- smoke 可以并行多个候选。
- 每个 smoke 只验证一个主候选，或一个明确声明的组合候选。
- dev/formal 不允许未记录的混合机制。
- formal 前不得事后改 gate。
- 结果影响论文或冻结时必须触发 `decisive-result-audit-zh`。
- 机制堆叠、连续失败、创新点说不清时必须触发 `research-iteration-audit-zh`。

### 输出：Smoke 排序表

```markdown
| 候选ID | smoke 命令 | 输出路径 | 主指标变化 | runtime | 异常 | 判定 | 下一步 |
|---|---|---|---|---|---|---|---|
| C1 | ... | ... | ... | ... | ... | PASS / CAUTION / BLOCK | dev / 继续 smoke / 放弃 |
```

### two-loop 参考边界

外部 `autoresearch` / two-loop 只作为节奏参考：

- Inner loop：多个候选快速 smoke。
- Outer loop：周期性复盘候选池，决定继续、合并、淘汰或换方向。

禁止吸收：

- “never stop”式无限自治。
- 不经用户或 gate 自动推进。
- 用快速实验替代 formal。
- 用叙事替代 raw/summary/metric 审计。

## 9. Step 6：Dev / Formal / Result Audit

主 skill：

```text
research-experiment-ops-zh
```

审计 skill 条件触发：

```text
decisive-result-audit-zh
research-iteration-audit-zh
```

### 做什么

把通过 smoke 的候选推进到 dev 和 formal。

```text
选定主候选
  -> dev 少量 run 验证趋势
  -> dev gate
  -> formal 固定配置正式跑
  -> summary
  -> decisive result audit
  -> iteration decision
```

### Dev / Formal 约束

- dev 前必须写清主候选 ID。
- dev 前必须锁定配置、指标、run、结果目录。
- formal 只收集，不再改公式。
- formal 后如果要写论文或冻结，必须读 raw、collector、metric、summary。
- 只有 summary 没有 raw 时，formal claim 直接 BLOCK。

### 输出：正式结果审计入口表

```markdown
| 项 | 路径 / 值 | 状态 |
|---|---|---|
| raw 结果 | ... | OK / MISSING |
| summary | ... | OK / MISSING |
| metric 实现 | ... | READ / NOT READ |
| 配置 | ... | OK / DIFF |
| 候选 ID | ... | OK |
| gate 标准 | ... | LOCKED / CHANGED |
```

## 10. Step 7：Iteration Decision

主 skill：

```text
research-iteration-audit-zh
```

注意：这里如果用户只是问“这轮要不要继续”，`research-iteration-audit-zh` 可以成为主 skill；如果是在实验执行过程中被触发，则作为审计 skill。

### 做什么

根据结果决定候选状态。

候选状态：

```text
promote：进入 dev/formal 或论文主创新
hold：保留但需要补证据
merge：与其他候选组成明确组合候选
drop：放弃并写入经验
block：禁止继续投入
```

### 输出：候选决策表

```markdown
| 候选ID | 当前证据 | 问题 | 决策 | 原因 | 下次继承约束 |
|---|---|---|---|---|---|
```

## 11. Step 8：Paper Memory / Writing

主 skill：

```text
paper-writing-zh
```

最多辅助：

```text
paper-memory-builder
research-statistics-reporting-zh
```

审计 skill 条件触发：

```text
decisive-result-audit-zh
```

### 做什么

论文写作前，把候选创新点按证据等级分层，不允许把未验证候选包装成贡献。

创新点分级：

```text
主创新点：formal 证据支持，可进入摘要/方法/贡献
次创新点：dev 或消融支持，只能谨慎写
失败候选：写入局限或未来工作，不能包装成贡献
放弃候选：进入经验记录
```

### paper-memory-builder 输出

外部 `paper-memory-builder` 只作为辅助，用于整理：

```text
.paper/claims.yml
.paper/figures.yml
```

`.paper/claims.yml` 最小字段：

```yaml
claims:
  - id: C1
    text: ""
    innovation_id: ""
    evidence_artifacts: []
    figure_or_table: []
    status: supported  # supported | draft | rejected | gap
    risk: ""
    allowed_claim_scope: ""
```

`.paper/figures.yml` 最小字段：

```yaml
figures:
  - id: Fig1
    file: ""
    key_numbers: []
    supports_claims: []
    caption_anchor: ""
```

### Paper Writing 检查点

- [ ] 是否先做 decisive result audit。
- [ ] 是否区分主创新、次创新、失败候选、放弃候选。
- [ ] 每条 claim 是否有 evidence artifacts。
- [ ] `gap` claim 是否没有写成正式结论。
- [ ] 图表描述是否与真实输出一致。

## 12. Step 9：Handoff / Archive

主 skill：

```text
project-handoff-zh
```

或：

```text
artifact-curator-zh
```

根据任务目标二选一，不同时作为主 skill。

### 做什么

把本轮科研状态整理到下一轮可以继承的状态。

必须输出：

```text
当前可信结果
stale 结果
已失败路线
候选池状态
主候选状态
脚本/命令
结果路径
本轮经验
下次必须继承的约束
下一步优先级
```

## 13. 外部候选 skill 策略

第一批候选外部 skill 只允许精选，不全量导入。

| 候选 | 用途 | 本地定位 |
|---|---|---|
| `literature-triage-matrix` | 文献比较矩阵 | 辅助 gap 判断 |
| `research-design-helper` | 多痛点/多假设设计模板 | 辅助研究设计 |
| `paper-memory-builder` | claim/figure 记忆 | 辅助论文写作 |
| `research-context-compressor` | `.research/` manifest | 可选补充，不替代六段式记忆 |
| `research-project-orienter` | 项目快速接班 memo | 可选补充 `project-handoff-zh` |

暂不直接纳入：

- `autoresearch`：自治原则冲突，只吸收 two-loop 思路。
- `academic-research-suite`：体量过大，只吸收 pipeline / integrity gate 思路。
- 大量模型训练工具 skill：遇到具体模型任务时再单独审计。

## 14. 外部 skill 冲突审计模板

任何外部 skill 后续安装或本地化前，必须填写：

```markdown
# 外部 Skill 冲突审计 - <skill-name>

## 基本信息
- 来源 zip / repo：
- 原始 skill 名：
- 许可证：
- 是否允许再分发：
- 拟纳入方式：安装 / 本地化改写 / 只吸收流程 / 不纳入

## 冲突审计
| 审计项 | 结果 | 说明 |
|---|---|---|
| 名称冲突 | PASS / CAUTION / BLOCK | 是否与现有本地 skill 同名 |
| 职责冲突 | PASS / CAUTION / BLOCK | 是否试图替代本地主 skill |
| 候选机制冲突 | PASS / CAUTION / BLOCK | 是否要求一次混改多个机制并直接 claim |
| 记忆冲突 | PASS / CAUTION / BLOCK | 是否替换六段式记忆 |
| 执行冲突 | PASS / CAUTION / BLOCK | 是否替代 smoke/dev/formal |
| 审计冲突 | PASS / CAUTION / BLOCK | 是否绕过结果审计和迭代审计 |
| 输出冲突 | PASS / CAUTION / BLOCK | 是否破坏 skill-outputs 规范 |
| 许可证冲突 | PASS / CAUTION / BLOCK | 许可证是否允许本地使用/改写/再分发 |

## 裁决
- 最终结论：PASS / CAUTION / BLOCK
- 可吸收部分：
- 禁止吸收部分：
- 本地上位规则：
```

裁决优先级：

```text
本地已验证规则
> 用户固定边界
> 多候选可比较性
> 结果可复现性
> 外部自动化效率
```

## 15. 可复制调用模板

### 15.1 新方向探索

```text
按科研 Skill 调用流 v3 执行新方向探索。
主 skill 使用 research-coach-zh；辅助 skill 最多使用 literature-triage-matrix 和 info-evidence-chain-zh。
先做 Memory Intake，生成项目状态卡、文献比较矩阵、候选痛点/假设/创新点列表。
然后做多候选 Gap Gate，输出候选池表，每个候选给 PASS / CAUTION / BLOCK。
不要进入实验执行。
```

### 15.2 多候选 smoke

```text
按科研 Skill 调用流 v3 执行候选 smoke。
主 skill 使用 research-experiment-ops-zh；辅助 skill 最多使用 agent-wait-monitor-zh 和 artifact-curator-zh。
每个候选必须独立 smoke，或明确声明组合候选 ID。
输出 smoke 排序表，选择 1 个主候选进入 dev。
如果结果影响论文、冻结或 claim，触发 decisive-result-audit-zh。
```

### 15.3 论文写作前 claim 整理

```text
按科研 Skill 调用流 v3 做论文写作前整理。
主 skill 使用 paper-writing-zh；辅助 skill 最多使用 paper-memory-builder 和 research-statistics-reporting-zh。
先触发 decisive-result-audit-zh 检查 formal/gate 结果。
把创新点分为主创新点、次创新点、失败候选、放弃候选。
生成 .paper/claims.yml 和 .paper/figures.yml 的内容草案。
没有 formal 或审计支持的 claim 只能标记为 draft 或 gap，不能写成正式贡献。
```

## 16. 验收测试

用三类样例验证本流程：

### 样例 A：新方向探索

- [ ] 只使用 1 个主 skill。
- [ ] 生成多个痛点、多个假设、多个创新点。
- [ ] 生成文献比较矩阵。
- [ ] 通过 Gap Gate 排序。
- [ ] 没有进入实验执行。

### 样例 B：算法实验

- [ ] 只使用 1 个主 skill。
- [ ] 辅助 skill 不超过 2 个。
- [ ] 多个候选可以并行 smoke。
- [ ] 每个 smoke 有候选 ID。
- [ ] dev/formal 只进入一个主候选或明确组合候选。
- [ ] 结果影响 claim 时触发审计。

### 样例 C：论文写作

- [ ] 先完成结果审计。
- [ ] 创新点分为主创新、次创新、失败候选、放弃候选。
- [ ] `.paper/claims.yml` 中每条 claim 有状态。
- [ ] 没有把未验证候选写成正式贡献。
- [ ] 六段式记忆被更新。

## 18. 最短记忆版

```text
多痛点 / 多假设 / 多创新点
  -> 文献和历史经验筛选
  -> 每个候选独立 smoke
  -> 只选一个主候选进 dev/formal
  -> formal 后审计 claim
  -> 论文只写证据支持的创新点
```
