# Web 设计工作流

> 一句话说明：把“让 Codex 直接写网站容易丑”的问题拆成设计规范、审美守门、工程实现、真实验收四个阶段，形成可复用的 Web UI 交付链路。

## 前置条件

- 本地已有相关压缩包：`web-design-main.zip`、`taste-skill-main.zip`、`skills-main.zip`。
- 本地已有工程技能：`skills/agent-skills-ao/frontend-ui-engineering/SKILL.md`。
- 新增编排技能：`skills/engineering/web-design-workflow-zh/SKILL.md`。

## 步骤

### Step 1: 设计前锁定规范

- **用什么**：`KAOPU-XiaoPu/web-design`
- **做什么**：先读 PRD、截图、参考 URL、品牌词或现有页面，生成项目根目录 `DESIGN.md`。
- **输入**：用户需求、项目文档、参考物、现有 assets。
- **输出**：9 章节 `DESIGN.md`。
- **完成标准**：颜色、字体、组件、布局、动效、响应式、Do/Don't 都能直接约束代码。

### Step 2: 编码前做审美守门

- **用什么**：`Leonxlnx/taste-skill`
- **做什么**：输出 Design Read，设置 `DESIGN_VARIANCE`、`MOTION_INTENSITY`、`VISUAL_DENSITY`，检查是否有 AI 味。
- **输入**：`DESIGN.md`、用户目标、参考风格。
- **输出**：确认后的设计方向和需要避免的反模式。
- **完成标准**：不是紫蓝渐变、三列卡片、米色高级感等默认模板；至少有一个能解释的 signature moment。

### Step 3: 工程化实现

- **用什么**：`addyosmani/frontend-ui-engineering`
- **做什么**：按项目真实技术栈实现组件、布局、状态、响应式和可访问性。
- **输入**：`DESIGN.md`、项目代码、依赖文件、现有组件。
- **输出**：可运行的页面或应用改动。
- **完成标准**：沿用技术栈；组件职责清楚；loading/empty/error/hover/focus/mobile 都完整。

### Step 4: 验收与修正

- **用什么**：Vercel Web Design Guidelines。
- **做什么**：读取最新 `command.md`，结合本地验收清单检查 UI、UX、Accessibility、Responsive、Performance。
- **输入**：运行中的页面、浏览器截图、控制台、构建结果。
- **输出**：修正后的页面和验证记录。
- **完成标准**：桌面和移动端视口真实检查通过；无明显重叠、溢出、控制台错误和键盘访问问题。

## 检查点

- [ ] 没有 `DESIGN.md` 时，不从零直接写完整网站。
- [ ] 编码前已经写出 Design Read。
- [ ] 代码中的颜色、字体、间距和动效能回溯到 `DESIGN.md`。
- [ ] 真实浏览器至少检查桌面和移动端。
- [ ] 最终回复包含路径、预览 URL、验证命令和剩余风险。

## 异常处理

| 异常场景 | 处理方式 |
|---|---|
| 用户只要求修一个 UI bug | 沿用现有设计规范，只局部修，不强行重做 `DESIGN.md` |
| 没有参考物且风格分叉明显 | 只问一个关键问题，例如“偏 Linear 克制还是 Awwwards 实验？” |
| Vercel 指南无法联网读取 | 使用 `references/acceptance-checklist.md` 降级验收 |
| 原项目没有框架 | 用最小可行静态 HTML/CSS/JS 或 Vite，不引入重栈 |
| 用户素材不足 | 用真实可替换占位图或明确请求素材，不用纯色块伪装图片 |
