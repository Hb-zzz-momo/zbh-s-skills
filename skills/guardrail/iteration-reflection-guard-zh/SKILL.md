# 迭代反思守门

## 目标

把“想到一个方案就执行”改成“先吸取历史、再提出单主假设、再反向审查、再自我修正、最后沉淀经验”的闭环。该技能是流程守门，不替代具体领域技能；科研任务可与 `research-coach-zh`、`research-experiment-ops-zh`、`research-iteration-audit-zh` 联用。

## 工作流

### 1. 迭代前先吸取历史

在提出计划前，先读取或回顾可用历史材料，至少形成四类清单：

- 已经失败的路线：明确哪些方向不能再走。
- 已经成立的证据：哪些结论可以继承，不要重复验证。
- 当前硬约束：项目规则、冻结口径、指标 gate、文件维护要求。
- 本轮可用缺口：历史没有否定、且仍有创新空间的问题。

如果历史材料不完整，先基于已有上下文列出“暂定历史经验”，不要凭空声称已验证。

### 2. 提出计划时必须防重复

计划必须先回答：

- 本轮单主假设是什么。
- 它解决的缺口是什么。
- 它为什么不是历史失败路线的改名重试。
- 它的创新性在哪里。
- 它要改善哪个核心指标或细节指标。
- 它的最小验证方式和止损条件是什么。

禁止一次提出多个互相独立的新机制；如果计划必须依赖多个机制，先拆成只读预审或降级。

### 3. 计划后列出五个致命痛点

每个新计划提出后，必须主动列出且只列出五个最致命痛点。每个痛点包含：

- 痛点描述：为什么它可能击穿方案。
- 击穿方式：会在哪个指标、数据集、工程约束或论文叙事上失败。
- 否决条件：出现什么证据就应停止。
- 修正方向：能否通过改计划解决；若不能，必须止损。

痛点要尖锐，不要写成普通风险提示。

### 4. 自我迭代 3-5 轮后再定稿

针对五个痛点修订计划。**不能只做一轮痛点分析后直接总结**；必须围绕同一个单主假设连续做 3-5 轮自我迭代，再给出定稿或止损结论。

每一轮都必须包含：

- 本轮针对上一轮哪些痛点做了修订。
- 哪些痛点被消除、降级或仍然保留。
- 修订后是否引入了新的致命痛点。
- 本轮三色判断和进入下一轮的理由。

每轮修订后做三色判断：

- 绿灯：痛点已被证据或约束合理处理，可以进入下一步。
- 黄灯：可继续，但必须先补一个只读预审或最小验证。
- 红灯：不能继续执行；必须换方向、降 claim 或请求关键证据。

默认在同一答复中做 **3 轮** 自我迭代；若第 3 轮后仍是黄灯且存在明确修正空间，可继续到第 4-5 轮。第 5 轮后必须定稿或止损，不得继续美化方案。

提前停止条件：

- 第 1-2 轮出现不可绕过的硬红灯时，可以提前止损，但必须明确说明“未满 3 轮的原因”。
- 如果用户明确要求只要简短结论，可以压缩表达，但仍要保留 3 轮修订逻辑。
- 如果进入执行会违反项目规则、伦理边界或实验 gate，必须止损，不能为了凑满轮数而提出伪修订。

### 5. 迭代完成后主动总结经验

每次迭代、实验、方案审查或流程闭环结束后，主动回答“这一轮可以总结什么经验”。总结至少包含：

- 这次真正学到了什么。
- 哪些假设被支持，哪些被否定。
- 下次不能再走的错路。
- 后续计划必须继承的新约束。
- 是否需要写入项目文档、Tracing、报告或 skill。

不要只总结结果，要总结可迁移的判断规则。

### 6. 五轮停滞必须深度复盘

如果同一目标连续 **5 轮迭代** 没有实质效果，必须暂停继续提出新候选，先做一次“停滞复盘”。满足任一条件即视为没有实质效果：

- 核心指标未达到预设 gate 或采纳门槛。
- 只有单 seed、单场景、局部标签收益，综合指标不稳定。
- 每轮都在同一类小修小补上徘徊，未改变瓶颈排序。
- 新方案反复被历史失败原因击穿。
- 结果虽偶有正向，但无法解释机制或无法复现。

停滞复盘必须输出：

- 最近 5 轮的候选、数据、失败原因和共同模式。
- 哪些方向已应判定为收益枯竭，不再重复。
- 当前最可信的瓶颈证据链，而不是直觉。
- 下一步最需要补的证据、工具、压测、profile 或理论分析。
- 继续优化、换方向、冻结成果、重构测量体系四者中的明确选择。

停滞复盘完成前，不得继续扩大实验或新增机制。

## 硬性规则

- 不读历史就不提出正式计划。
- 不列五个致命痛点就不进入执行。
- 不完成 3 轮自我修订就不定稿；除非出现不可绕过硬红灯，并明确说明提前止损原因。
- 第 4-5 轮只能用于处理仍有修正空间的黄灯，不得为了美化红灯方案而拖延止损。
- 红灯未解除不新增机制、不启动实验、不扩大任务。
- 历史失败路线不能换名字重试。
- 迭代结束必须沉淀经验，而不是只报告完成。
- 连续 5 轮没有实质效果时，必须先做停滞复盘和方向重选，再进入下一轮。
