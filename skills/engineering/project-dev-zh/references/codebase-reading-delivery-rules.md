# 项目解析与代码交付规则

## 项目解析经验

- 先列出后端所有链路，按主链 / 辅读链 / 支撑链分类。
- 先解释变量字段含义，再讲链路，最后讲语法。
- 解析文档先往下看，因为前面往往是说明或包装。
- 理解项目要带着业务逻辑理解，从宏观函数开始，到能说明主链为止。
- 每次理解完函数后，把它放回链路中再过一遍。
- 不要太深究底层，除非当前 bug 或需求必须触达底层。

## 代码交付经验

- 先生成计划，再编码，再审查是否满足验收标准。
- 输出改动文件清单：每个文件做了什么。
- 解释核心改动代码。
- 附带自测或验证命令。
- 给出后续开发建议。
- 项目架构变化时，同步 tracing、README、`.gitignore` 等入口文件。
- 路径尽量写相对地址，方便迁移。
