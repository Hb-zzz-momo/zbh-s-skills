# Web Design Workflow Modes

## 决策表

| 问题 | 选择 |
|---|---|
| 只需 1-7 天内展示、课程或比赛 Demo？ | 快速原型 |
| 需要正式域名、真实用户、长期维护？ | 正式产品 |
| 已有可运行页面，只是丑或不统一？ | 现有改版分支 |
| 用户明确要求先看视觉稿或高度视觉化？ | 图像先行分支 |
| 只修一个按钮、溢出、焦点或交互 bug？ | 局部 UI Bug，不走完整链 |

## 快速原型模式

### Step 1: 一页 Brief

- **输入**：一句话目标、受众、核心页面、主要功能、参考风格。
- **输出**：项目内 brief 或聊天中锁定的边界。
- **完成标准**：能说清“给谁、解决什么、演示什么”。

### Step 2: DESIGN.md Lite

- **输入**：Brief、参考图/URL、现有素材。
- **输出**：精简但完整的 `DESIGN.md`。
- **完成标准**：颜色、字体、布局、组件、动效、禁用项和响应式都有可执行规则。

### Step 3: 纵向切片

- **输入**：DESIGN.md。
- **输出**：Hero + 首屏内容，或一个完整核心流程。
- **完成标准**：真实内容、响应式和主要交互状态可见。

### Step 4: 扩展首版

- **输入**：通过自检的切片。
- **输出**：可运行原型。
- **完成标准**：核心页面和主路径可演示，代码没有明显大文件和重复结构。

### Step 5: 浏览器验收

- **输入**：本地 dev server。
- **输出**：桌面/移动截图和问题修复。
- **完成标准**：无控制台错误、溢出、遮挡和主要交互故障。

### Step 6: 交付

- **输入**：通过验收的原型。
- **输出**：本地 URL；按需提供 Preview Deployment。
- **完成标准**：用户能直接打开和演示。

## 正式产品模式

### Step 1: 产品定义

输出 PRD、用户流、页面结构、数据来源和验收标准。

### Step 2: 设计规范

输出正式 `DESIGN.md`、组件状态、内容策略、素材清单和响应式策略。

### Step 3: 技术方案

沿用现有技术栈；新项目按需求选择最小可维护方案。定义路由、组件边界、数据边界、测试和部署方式。

### Step 4: 纵向切片

完成一条从 UI 到数据、状态和测试的真实路径。通过后再扩展。

### Step 5: 增量构建

按页面或用户流小步实现，每个切片运行可用的 lint、typecheck 和测试。

### Step 6: 设计与浏览器验收

按 DESIGN.md、真实视口、键盘、控制台、网络和交互状态验收。

### Step 7: Code Review

检查改动范围、组件边界、重复、依赖、安全、性能和回归测试。

### Step 8: Preview Deployment

生成可访问预览，使用真实设备或浏览器复验关键路径。

### Step 9: 上线门禁

执行项目定义的性能、Accessibility、SEO、安全、监控和回滚检查。不要机械套固定分数，阈值由项目要求决定。

### Step 10: 发布与迭代

正式部署，记录监控、反馈、问题和下一轮小步改动。

## 阶段门禁

| Gate | 必需证据 | 未通过处理 |
|---|---|---|
| G0 需求门 | Brief/PRD 能回答受众、目标、范围 | 补齐最关键缺口 |
| G1 设计门 | DESIGN.md + Taste Gate | 修改规范，不写代码 |
| G2 切片门 | 真实浏览器截图 + 核心交互 | 只返工切片 |
| G3 构建门 | lint/typecheck/test/build 可用项通过 | 修复后再扩展 |
| G4 验收门 | 桌面/移动、键盘、控制台、网络 | 修复并复验 |
| G5 发布门 | Preview、性能/SEO/安全/回滚证据 | 不发布正式环境 |

## 异常处理

| 场景 | 处理 |
|---|---|
| 没有素材 | 使用真实可替换占位资源并列出待替换清单，不用纯色块冒充图片 |
| 风格信息不足 | 给出 2-3 个差异明确方向；能合理判断时直接选定并记录 |
| 用户要求全自动 | 不等待逐阶段确认，保留 Gate 自检后继续 |
| 用户要求复刻网站 | 区分设计语言参考与 1:1 复制，避免侵犯品牌和版权 |
| 构建环境不可用 | 完成静态检查并明确未验证项，不宣称已通过 |
