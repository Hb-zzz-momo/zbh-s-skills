# 接班文档写作规范

## 三份主文档的边界

### 项目统一认知

用途：统一项目全局判断。

应写：

- 项目一句话定义
- 当前正式主线
- 平台/代码库角色
- 论文或产品 claim
- 当前冻结版本
- 适用边界
- 不该误解的内容

不写：

- 每次实验的细节
- 每次运行进度
- 长表格结果
- 临时调参过程

### AI接班手册

用途：让下一位 AI 快速接手。

应写：

- 推荐阅读顺序
- 当前有效事实
- 主资产目录
- 核心源码入口
- 当前冻结版和开发分支
- 当前可信结果与 stale 结果
- 下一步优先级
- 协作规则和文档更新规则

不写：

- 全量历史复述
- 每一轮实验日志
- 无法行动的背景故事

### 工作流细化

用途：维护整体实验/开发链路。

应写：

- 阶段划分
- 入口脚本
- 数据集/算法/指标口径
- 验收标准
- 汇总路径
- 异常处理路径

不写：

- 当前跑到第几个 run
- 某次临时失败
- 单次问答

## 更新规则

- 全局认知变化：更新 `项目统一认知.md`、`AI接班手册.md`、Tracing。
- 整体实验链路变化：更新 `工作流细化.md`、`AI接班手册.md`、Tracing。
- 当前进度或结果变化：更新 Tracing 和对应 step/report，不必改主工作流。
- 新问题或修复经验：更新 `Tracing/问题记录.md`。
- 接班默认事实变化：更新 `AI接班手册.md`。
- skill 目录新增、删除、重命名，或 `SKILL.md` 的 `name/description` 变化：同步更新 `skills/README.md`。
- 项目根目录结构、入口命令或忽略规则变化：同步更新 README、`.gitignore` 或对应说明文档。

## 接班状态补丁格式

建议在文档靠前位置追加：

```markdown
## 当前接班状态补丁（YYYY-MM-DD）

- 当前冻结主线是 `...`。
- 当前最好开发分支是 `...`，但不是冻结版。
- 已证明不值得继续的方向是：...。
- 当前最优先下一步是：...。
- 新 AI 先读：...。
```

## 最小交接清单

交接文档至少要让新 AI 回答这些问题：

- 项目主任务是什么？
- 正式主底座在哪里？
- 当前主问题类或主模块是什么？
- 当前冻结算法/版本是什么？
- 当前最好但未冻结的开发分支是什么？
- 哪些结果可信，哪些结果 stale？
- 下一步先做什么？
- 哪些方向不要再做？
- 如果整体链路变化，应更新哪些文档？

## Tracing 最低标准

- `修改记录.md`：记录改了什么、为什么改、影响范围。
- `问答记录.md`：记录用户问题、答复结论和关键解释。
- `结论记录.md`：记录阶段性判断、通过/失败/止损原因。
- `问题记录.md`：记录问题背景、复现条件、解决方案和后续规避规则。
