# 学术论文撰写辅助技能

## 技能目标

以"材料驱动、阶段递进、严格区分事实与推测"的方式辅助学术论文各章节的撰写，确保输出可直接用于投稿。

## Markdown 交付要求

1. 使用本技能完成正式输出时，除聊天中的简要说明外，必须在当前工作区新建一个 Markdown 文档保存完整结果。
2. 默认保存位置为当前工作区 `skill-outputs/paper-writing-zh/`（与 `artifact-curator-zh` 按技能分子目录一致；根目录为 **legacy-flat** 兼容）。若该子目录不存在须先创建。
3. 默认文件名格式为 `中文主题_YYYYMMDD_HHMMSS.md`（置于上述目录下；文件名不再重复 skill 名）；若主题不明确，使用 `结果_YYYYMMDD_HHMMSS.md`。
4. Markdown 文档必须包含完整章节正文、结构说明、修改建议与自检结果（如有）。
5. 若用户明确指定保存路径或文件名，以用户要求为准；若用户明确要求只在对话中回答，可跳过创建文件。
6. 最终回复中要说明新建的 Markdown 文件路径（须含 `skill-outputs/paper-writing-zh/` 前缀），以及文件里包含的主要内容。

## 核心原则

1. **材料驱动**——所有论述必须基于用户提供的实验数据、代码或文献，不凭空编造
2. **事实/推理/建议三分**——明确区分已有证据、由证据推出的结论、以及个人建议
3. **阶段递进**——识别当前写作阶段，按该阶段的规范执行
4. **不造假**——不编造实验数据、不虚构引用文献、不夸大实验结果

## 论文章节写作流程

### 阶段识别

根据用户请求判断当前写作阶段：

| 阶段 | 章节 | 核心要求 |
|------|------|----------|
| 1 | 摘要（Abstract） | 问题→方法→结果→结论，4句话框架 |
| 2 | 引言（Introduction） | 大背景→具体问题→现有方法不足→本文方案→贡献列表 |
| 3 | 相关工作（Related Work） | 按主题分组综述，指出每组的gap |
| 4 | 方法（Methodology） | 问题形式化→整体框架→各模块详述→算法伪代码 |
| 5 | 实验（Experiments） | 设置→数据集→基准→指标→结果→分析 |
| 6 | 结论（Conclusion） | 总结贡献→局限性→未来工作 |

### 各阶段详细规范

#### 摘要

- 4句话结构：background → problem → method → result
- ≤250词（或会议要求的字数限制）
- 不出现"本文首次""创新性地"等主观断言（除非数据支持）
- 关键数值必须来自实际实验

#### 引言

- 漏斗结构：宽→窄→本文
- 第1段：领域大背景 + 重要性
- 第2段：具体问题 + 现有方法
- 第3段：现有方法的不足（每条不足对应后面的一个贡献）
- 第4段：本文方案概述
- 第5段：贡献列表（与第3段的不足一一对应）

#### 相关工作

- 按主题/方法类别分组，不按时间流水账
- 每组末尾指出该类方法的 gap/limitation
- gap 应自然过渡到本文方案的设计动机
- 引用格式严格按目标期刊/会议要求

#### 方法

- 先写问题形式化（符号定义、优化目标）
- 再写整体框架图描述
- 然后逐模块详述（输入→处理→输出）
- 公式编号连续，每个公式后附文字解释
- 算法伪代码放在方法末尾

#### 实验

- 实验设置：数据集描述、对比方法列表、评价指标定义、实现细节
- 结果分析：先整体排名，再逐指标/逐数据集分析
- 消融实验：每个模块的贡献
- 参数敏感度：关键超参的影响
- 所有数值必须来自实际实验运行

#### 结论

- 用1-2段总结贡献（不重复摘要的原话）
- 诚实陈述局限性
- 给出1-2个可行的未来工作方向

## 写作质量约束

### 语言规范

1. 学术正式语体，不口语化
2. 避免绝对化表述（"always""never"→ "typically""rarely"）
3. 避免第一人称过度使用（"We propose" 可以，"I think" 不行）
4. 段落间有过渡句

### 事实/推理/建议标注

当内容可信度不同时，使用以下标注帮助用户识别：

- **[事实]**：来自实验数据或已发表文献
- **[推理]**：由事实逻辑推出，但未直接验证
- **[建议]**：个人经验性建议，需用户判断

### 自检清单

写作完成后执行：

- [ ] 是否有未标注来源的数据/结论？
- [ ] 贡献列表与引言中的不足是否一一对应？
- [ ] 公式符号是否全部在首次出现时定义？
- [ ] 是否有前后矛盾（摘要说A，实验说B）？
- [ ] 参考文献格式是否统一？

## 润色模式

当用户说"润色"时：

1. 先通读全文，标出问题类型：
   - 语法错误
   - 逻辑不通
   - 表述模糊
   - 前后不一致
   - 格式问题
2. 按问题严重度排序
3. 给出修改建议（原文 → 修改后），附修改理由
4. 不大改结构（除非用户要求）

## 约束

1. 不编造引用文献——用户未提供的文献用 [?] 占位
2. 不虚构实验数据——用户未提供的数据用 [待补充] 标注
3. 不替用户做学术判断——方案选择、理论创新点由用户确认
4. 输出默认英文（论文正文），分析说明用中文

## 自我迭代模式

### 触发条件

当用户在使用本技能后给出反馈（如"这部分不好用""输出格式需要改""多了/少了XX"），或主动说"优化这个skill"时，进入自我迭代模式。

### 迭代工作流

1. **收集反馈**：明确用户不满意的具体环节（哪个模式 / 哪个步骤 / 什么问题）
2. **诊断根因**：是规则缺失、规则冲突、粒度不够、还是场景未覆盖？
3. **提出修改方案**：列出拟修改的条目（原文 → 修改后），说明修改理由
4. **用户确认**：修改方案经用户确认后才执行
5. **写入 SKILL.md**：将修改直接应用到本技能文件
6. **记录迭代日志**：在 `references/iteration-log.md` 追加本次迭代记录

### 迭代日志格式

```markdown
## [日期] 迭代 #N

- **触发反馈**：...
- **根因**：...
- **修改内容**：...
- **影响范围**：模式X / 步骤Y
```

### 迭代约束

- 不破坏现有模式结构——增量修改，不推倒重写
- 每次迭代只改一个关注点——不趁机大改
- 修改后 SKILL.md 总行数仍 < 500
- 重大结构变更（如新增/删除模式）必须用户明确同意
