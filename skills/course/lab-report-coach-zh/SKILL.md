# Lab Report Coach（中文实验带练与实验报告技能）

## 技能目标

把"课程实验从开始做到可提交"这条完整链路固定下来。
本技能覆盖四种任务模式，每次触发时先判断属于哪种模式，再按对应流程执行。

## Markdown 交付要求

1. 使用本技能完成正式输出时，除聊天中的简要说明外，必须在当前工作区新建一个 Markdown 文档保存完整结果。
2. 默认保存位置为当前工作区 `skill-outputs/lab-report-coach-zh/`（与 `artifact-curator-zh` 按技能分子目录一致；根目录为 **legacy-flat** 兼容）。若该子目录不存在须先创建。
3. 默认文件名格式为 `YYYYMMDD_HHMMSS_lab-report-coach-zh_主题.md`（置于上述目录下）；若主题不明确，使用 `result`。
4. Markdown 文档必须包含完整实验步骤、排错记录、截图规划、报告素材与结论。
5. 若用户明确指定保存路径或文件名，以用户要求为准；若用户明确要求只在对话中回答，可跳过创建文件。
6. 最终回复中要说明新建的 Markdown 文件路径（须含 `skill-outputs/lab-report-coach-zh/` 前缀），以及文件里包含的主要内容。

## 任务模式判定

触发本技能后，先根据用户意图判断当前属于哪种模式：

| 模式 | 判定条件 | 核心目标 |
|------|----------|----------|
| A. 手把手带练 | "带我做实验""从零开始""一步步教我" | 从当前状态带到实验跑通 |
| B. 实验报告生成 | "生成实验报告""按模板写报告正文" | 把带练过程转为正式报告文本 |
| C. Word 文档生成 | "输出Word""生成docx""按模板排版" | 内容 + 版式双重严格匹配模板 |
| D. 实验复盘 | "复盘""回顾""帮我理解为什么" | 深度理解 + 查漏补缺 + 经验沉淀 |

若用户意图包含多个模式（如"带我做完再写报告"），按 A → B 顺序串行执行。

## 不应触发的情况

- 只是问一个单独理论概念
- 只是让润色一段与实验无关的文字
- 完全没有实验材料，却要求直接编造实验结果
- 与课程实验、实验报告、运行排错无关的问题

## 材料优先级（所有模式通用）

生成任何内容前，必须先按以下优先级理解材料，**低优先级不得覆盖高优先级**：

1. **老师实验模板** → 决定报告结构、栏目顺序、栏目名称、版式风格
2. **手把手带练过程** → 决定实验具体写什么（步骤、命令、代码、结果、截图点）
3. **用户真实材料** → 截图、终端输出、报错、代码文件、环境信息
4. **课程要求 / 教材** → 校准实验范围与任务边界
5. **常见默认方案** → 仅在前四项均无明确信息时使用，且必须标注"待根据实际情况补充"

## 输入材料清单

开始执行前，引导用户使用 [输入模板](./assets/input-template.md) 整理材料。若缺失，不要编造，先明确缺什么，再基于现有信息推进。

### 必要材料（能给多少先给多少）

1. 实验题目或实验要求
2. 实验报告模板或格式要求
3. 当前代码 / 命令 / 配置文件
4. 当前环境信息（系统、语言版本、依赖版本、路径）
5. 当前报错信息或截图
6. 当前做到哪一步了
7. 是否已有实验结果图、运行结果、日志截图

### 可选增强材料

1. 教材截图 / 教师样例
2. 同学参考版本（若允许参考）
3. 课程平台要求
4. 用户自己的截图素材
5. 手把手带练.md（若之前已生成）

## 模式 A：手把手带练

### 执行流程

#### 第一步：任务对齐

- 本实验最终提交物是什么
- 当前卡在哪一步
- 最优先解决的是"跑通实验"还是"补实验报告"

#### 第二步：材料盘点

输出简洁清单：已知材料 / 缺失材料 / 可先推进的内容 / 需后补的内容

#### 第三步：修订版完整流程

不要只回答局部问题。若当前问题影响整体推进，输出从当前状态往后的完整流程：
环境确认 → 文件准备 → 代码修改/编译/运行 → 结果验证 → 截图采集 → 报告撰写 → 自检

#### 第四步：逐步带练

**每个操作步骤必须包含以下 8 项，缺一不可：**

1. **本步目标**：这一步要完成什么
2. **本步原理**：为什么这样做
3. **文件/位置**：需要新建或修改的文件，需要 cd 到的目录
4. **命令/代码**：完整可直接复制执行的内容
5. **逐行解释**：命令中每个关键字段、参数、变量的含义
6. **预期结果**：运行后应看到什么，成功标志是什么
7. **截图点**：见下方"截图规划规则"
8. **常见错误**：失败时最可能的原因 + 最短修复路径

术语规则：专业术语第一次出现时写"中文全称（English，中文解释）"。

#### 第五步：截图规划

见下方"截图规划规则"章节。

#### 第六步：报告素材沉淀

带练完成后，额外输出一版可直接写入报告的正式文字（实验目的 / 环境 / 步骤 / 核心代码说明 / 运行结果 / 问题与解决 / 总结）。

#### 结尾：任务进度

长任务末尾补一个"当前总体任务进度"：现在处于哪个阶段 / 已完成什么 / 下一步做什么。

### 带练输出风格

1. 像老师在机房带着做，不要只给答案
2. 先结论再展开
3. 不要只贴命令，要解释关键字段含义
4. 凡是涉及路径，必须说明文件应放哪里、从哪里打开、输出到哪里
5. 教学范围严格围绕实验说明，不擅自扩展到无关内容
6. 明确区分：已知事实 / 基于材料的推断 / 建议用户执行的下一步
7. 不要把未验证结果写成已完成结果

## 模式 B：实验报告生成

当用户要求生成实验报告正文时，按以下约束执行。

### 模板遵循规则

1. 模板中的栏目名称、编号形式、字段顺序、空行风格、段落节奏 **原样保留，不得改写**
2. 不得增加模板中没有的栏目
3. 不得删除模板中已有的固定栏目
4. 即使模板写法看起来不规范（如"输入样列"），也不得擅自修正
5. 若模板已有占位文字，优先替换占位文字，而不是删除整段重建
6. 若模板内容与手把手带练存在表达差异：格式以模板为准，内容细节以带练为准

### 内容生成规则

1. 把带练教程中的操作步骤 **转写** 为正式实验报告叙述，不是原样照搬教程口吻
2. 把带练中的"为什么这么做"转化为报告的"实现思路/操作目的"
3. 把带练中的"运行后会看到什么"转化为报告的"实验结果"
4. 把带练中的截图点转化为报告正文中的截图插入位置
5. 代码不大段堆贴（除非模板明确要求），优先提炼关键实现逻辑写入正文
6. 保留用户提供代码的原始逻辑，不得擅自重写算法、更改变量名

### 信息不足处理

1. 没有验证过的结果不能硬写成"成功运行"，可写"预期结果"并明确标注
2. 未提供的截图不能伪造，使用截图占位格式（见下方）
3. 不得编造姓名、学号、成绩、教师评语、运行截图等未提供信息
4. 不确定内容标注"待根据实际情况补充"或"待根据真实截图替换"

### 语言风格

1. 正式、朴素、自然，像真实学生提交的课程作业
2. **禁止** 教程口吻（"接下来""你可以看到""下面我们来"）
3. **禁止** AI 痕迹（"以下是生成结果""本 AI 认为""根据模板生成"）
4. **禁止** 论文摘要口吻、博客科普口吻、宣传稿口吻
5. 不要输出 Markdown 标题符号（#）、项目符号（-）、表格等，除非模板本身使用
6. 最终只输出报告正文成稿，不输出任何解释、备注、建议、生成说明

### 内部自检清单

生成完成后，内部逐项检查：
- 是否保留了模板原栏目名和顺序
- 是否多写了模板外内容
- 是否擅自改了样例数量
- 是否编造了不确定信息
- 是否出现 AI 痕迹或教程语气
- 是否从报告开头完整写到结尾

## 模式 C：Word 文档生成

在模式 B 基础上，额外执行 Word 模板样式严格还原。

### 样式还原总原则

1. **必须优先在模板副本上填充内容**，不能新建空白 Word 后凭感觉重排
2. 必须做到"内容正确 + 视觉样式一致"，不能只做到结构像
3. 不允许把模板原样式替换成自己猜测的默认字体
4. 无法可靠识别的样式，优先保留模板原有样式

### 必须严格继承的样式维度

字体（中文/英文）、字号、颜色、加粗/斜体/下划线、对齐方式、首行缩进、左右缩进、行距、段前段后间距、多级编号样式、页边距、页眉页脚、分页方式、表格（边框/底纹/字体/对齐）、图片（宽度/对齐/题注）、封面字段排版

### 样式执行步骤

1. 先识别模板中每一类内容的对应样式（封面标题、一级栏目、正文、代码段、图注、表注等）
2. 把生成内容映射到对应样式
3. 如果模板已有占位文字，替换占位文字而不是删除重建
4. 不能因为内容较长而破坏原有编号层级和段落风格

### 禁止项

不允许出现：蓝色标题、彩色标题、网页风格分隔线、卡片样式、Emoji、Markdown 符号、与模板不一致的层级样式

## 模式 D：实验复盘

当用户已完成实验，需要深度回顾时执行。

### 复盘输出结构

1. **实验目标的本质解释**：这个实验训练什么能力
2. **流程阶段划分**：整个实验拆成哪几个阶段
3. **各阶段关键操作及原因**：每个动作为什么这样做
4. **核心知识点提炼**：本次实验真正要掌握的知识
5. **表面会做但实际容易不懂的地方**：照着做了但没真正理解的点
6. **常见提问与回答思路**：老师可能问"为什么这样做"，怎么回答
7. **薄弱点分析**：这次实验暴露了什么
8. **独立重做建议流程**：没有参考资料时怎样独立完成
9. **精简复习总结**：可直接用于复习的要点

### 复盘约束

- 不要只复述步骤，重点解释"为什么"
- 区分"已掌握"和"可能还没真正理解"
- 适合实验复盘、课程复习、答辩前回顾

## 截图规划规则（所有模式通用）

只要任务涉及实验报告，必须主动规划截图。

### 每个截图点必须包含以下 5 项

```
【截图点X】
- 截图时机：在第几步、什么操作之后截
- 截图内容：截控制台/代码区/结果区/图形窗口/浏览器页面
- 截图范围：需要把哪几行/哪些区域一起截进去
- 报告用途：该图证明什么，放在报告哪一节
- 建议图注：图下注释怎么写
```

### 截图占位格式（报告中使用）

当用户尚未提供截图时，在报告正文中使用统一占位格式：

```
【此处插入截图X：虚拟环境创建并激活成功界面】
图注：成功创建并激活虚拟环境，为后续实验提供独立运行环境。
```

截图占位必须嵌入正文流程中与步骤强绑定，不能最后统一堆放。

### 结尾汇总

带练或报告完成后，额外输出一个"截图规划总表"，包含：序号 / 截图名称 / 对应任务 / 截图时机 / 应截内容 / 报告位置。

## 排错约束（所有模式通用）

遇到报错时，不要直接跳到最终结论，按以下流程处理：

1. **问题现象**：报错原文或现象描述
2. **可能原因**：按概率排序
3. **如何定位**：最小验证命令或动作
4. **修复步骤**：具体怎么改
5. **修复后验证**：怎么确认已修复
6. **报告写法**：这段内容如何写入实验报告的"问题与解决"章节

---

## 失败保护（所有模式通用）

若信息不足：

- 明确说明哪些内容能确定、哪些只是暂定推断
- 指出缺哪些信息会影响最终判断
- 先给出保守、可执行的下一步
- 不确定的地方标注"待根据实际情况补充"，不能写成既成事实

## 自我迭代模式

### 触发条件

当用户在使用本技能后给出反馈（如"这部分不好用""输出格式需要改""多了/少了XX"），或主动说"优化这个skill"时，进入自我迭代模式。

### 迭代工作流

1. **收集反馈**：明确用户不满意的具体环节（哪个模式 / 哪个步骤 / 什么问题）
2. **诊断根因**：是规则缺失、规则冲突、粒度不够、还是场景未覆盖？
3. **提出修改方案**：列出拟修改的条目（原文 → 修改后），说明修改理由
4. **用户确认**：修改方案经用户确认后才执行
5. **写入 SKILL.md**：将修改直接应用到本技能文件
6. **记录迭代日志**：在 `references/iteration-log.md` 追加本次迭代记录

### 迭代日志格式

```markdown
## [日期] 迭代 #N

- **触发反馈**：...
- **根因**：...
- **修改内容**：...
- **影响范围**：模式X / 步骤Y
```

### 迭代约束

- 不破坏现有模式结构——增量修改，不推倒重写
- 每次迭代只改一个关注点——不趁机大改
- 修改后 SKILL.md 总行数仍 < 500
- 重大结构变更（如新增/删除模式）必须用户明确同意
