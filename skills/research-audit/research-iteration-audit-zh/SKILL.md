# 科研迭代自审技能

## 技能目标

在每次算法迭代产出数据后，主动检测以下 7 类常见科研陷阱，输出红黄绿三灯判定和明确的下一步建议，防止研究者在错误方向上继续堆机制、累积研究债务。

核心问题域：

1. 机制堆叠（新增机制越来越多但贡献度不明）
2. 创新模糊与论文可读性不足（说不清原创点，也写不成审稿人能读懂的机制）
3. 性能退化（迭代多轮后仍然打不过 SOTA）
4. Runtime 膨胀（机制越多每 run 越慢）
5. 研究债务（承诺贡献与实测贡献差距拉大）
6. SOTA 架构落后（骨架错了再多补丁救不回）
7. 失败边界（连续同方向失败仍继续碰运气）

## 和其他 skill 的关系

- **research-coach-zh**: 负责"从零分析问题、设计算法、生成周报"——向前做
- **本技能**: 负责"迭代完成后的事后体检"——向后审
- **paper-writing-zh**: 负责论文撰写
- **workflow-forge-zh**: 负责流程抽象

当用户要"做新算法"→ `research-coach-zh`；当用户要"看看现在这版行不行"→ **本技能**。

## 触发时机（审计必须跑的 5 个场合）

| 场合 | 判定 | 说明 |
|------|------|------|
| 每次 smoke gate 产出数据后 | 必须 | 尤其是 smoke 出现负向时，立即审，不要直接进 dev |
| 每次 dev gate 产出数据后 | 必须 | 决定是否进 final，或回退 |
| 每次新增机制合并到主分支前 | 必须 | 预审：新增是否会把机制数推进红线 |

## 7 大审计维度及阈值

### 维度 1：机制堆叠度

**采集指标**：
- `N_mech`：当前算法并行运行的**独立机制数**（按"可独立开关"粒度计数）
- `AblationRegistry`：机制注册表完整度，每条需含「加入时预期贡献 %」+「最近一次消融实测贡献 %」
- `JointAblationDone`：是否跑过"裸核消融"——一次性关闭 ≥ 3 个机制看 HV 掉多少

**红灯 🔴**：
- `N_mech > 10` 且 `JointAblationDone = false`
- 或存在任何机制**从未被消融过**

**黄灯 🟡**：
- `N_mech ∈ [6, 10]`
- 或某机制单独消融贡献 < 2%

**绿灯 🟢**：
- `N_mech ≤ 5`
- 所有机制单独消融贡献 ≥ 3%
- 机制注册表完整

**判定要点**：
> "可独立开关"的判据：代码里是否存在一个 boolean 开关或参数 mode 能把这个机制完全关闭，而算法其余部分仍然正常运行。不能独立关闭的多步骤流程算 1 个机制。

---

### 维度 2：创新清晰度与机制论文可读性

**采集指标**：
- 每个机制的原创度打分（0 / 0.3 / 0.5 / 1）：
  - 0 = 直接继承已有工作（如 NSGA-II crowding、Sparse init from SparseEA）
  - 0.3 = 是已有技术的微变体
  - 0.5 = 在已有框架上的显著改进
  - 1.0 = 真正原创的新机制
- `OriginalityTotal` = 所有机制原创度之和
- `PaperHeroClarity`：能否一句话说出"删掉就致命的 1-3 个主角机制"
- `MechanismReadability`：机制能否写成论文读者可理解的"问题动机 → 核心设计 → 数学/伪代码定义 → 预期作用 → 消融证据"链条
- `TermEquationMapping`：机制名、代码变量、公式符号、伪代码步骤是否一一对应
- `ReviewerReconstructability`：审稿人不看代码时，是否能根据论文描述复现机制的关键流程和开关

**红灯 🔴**：
- `OriginalityTotal < 1.0`
- 或无法一句话说清主角
- 或宣称的创新点 ≥ 5 个但每个原创度都 < 0.5
- 或主角机制只能用代码改动解释，无法写成清晰的论文段落、公式/伪代码或机制图
- 或机制名、公式符号、代码变量互相脱节，审稿人无法判断机制到底改了什么

**黄灯 🟡**：
- `OriginalityTotal ∈ [1.0, 2.0]`
- 或主角机制的独立贡献未做过消融证明
- 或机制读起来像"工程技巧集合"，有动机但缺少正式定义、伪代码、复杂度说明或消融对应项

**绿灯 🟢**：
- `OriginalityTotal ≥ 2.0`
- 主角清晰（1-3 个）
- 每个主角的独立贡献已被消融证明
- 每个主角机制都有可放进论文方法节的正式表述：命名稳定、符号清楚、伪代码/公式可追踪、实验表能对应到该机制

**判定要点**：
> 用审稿人视角写出「如果审稿人只给你 5 分钟解释，你会说哪 3 个创新点，每个在文献中和什么对照」。说不出来就是红灯。

**论文可读性补充判据**：
> 一个机制即使实验有效，如果只能在代码里看懂、论文里说不清，也不能作为主创新点硬写。必须能用 1 段动机、1 个正式定义或伪代码、1 个作用解释、1 个消融证据完成闭环；否则只能降级为工程实现细节或附录说明。

### 维度 3：性能胜出度

**采集指标**：
- `WinGrid[i,j]`：指标 `i` × 数据集 `j` 上，我方是否**严格胜于任一 SOTA**
- `WinRate` = 胜出格子数 / 总格子数
- `ParetoFeasible`：在 (性能, runtime) 平面我方是否被 SOTA **支配**

**红灯 🔴**：
- 主指标（如 HV）胜出数据集 = 0
- 或 `ParetoFeasible = false`（runtime 更慢且性能更差）

**黄灯 🟡**：
- `WinRate < 50%`
- 或只在少数数据集上偶尔胜出

**绿灯 🟢**：
- 项目目标要求的"4/6 数据集 HV+IGD 同时胜出"达成
- 或整体 `WinRate ≥ 60%`

**判定要点**：
> 禁止只报均值排名，必须逐数据集逐指标呈现格子。禁止用"相当""接近""不差于"等模糊词，必须给具体数值。

---

### 维度 4：Runtime 预算

**采集指标**：
- 每代大 O 分析表：列出每个机制的复杂度项
- `DominantComplexity`：最高复杂度项（如 `O(N²·D)`）
- `ρ = T_mine / T_baseline`：实测 wallclock 比值（同硬件、同参数）
- `HighDimSafe`：在 D ≥ 5000 时是否还能在合理时间跑完

**红灯 🔴**：
- `ρ ≥ 3`
- 或存在 `O(N²·D)` 及以上项且无法解释为必要
- 或高维数据集 `D ≥ 5000` 下单 run > baseline 5 倍

**黄灯 🟡**：
- `ρ ∈ [2, 3)`
- 或存在 `O(N²)` 但未经优化

**绿灯 🟢**：
- `ρ < 2`
- Dominant complexity ≤ `O(N·D·log N)`

**判定要点**：
> Runtime 差不是工程质量问题，多数情况下是机制数量问题。每个机制都是额外复杂度，15 个机制就是 15 倍开销。

---

### 维度 5：研究债务

**采集指标**：
- `DebtTable`：每个机制的「加入时承诺贡献 %」vs「当前消融实测贡献 %」
- `AvgGap` = 平均差距
- `PerformanceVsMechanismCurve`：画出"迭代轮次 vs 机制数 vs 主指标"三折线，看机制数还在涨但主指标是否停滞

**红灯 🔴**：
- 连续 3 轮迭代机制数在涨但主指标停滞（或下降）
- 或 `AvgGap > 5%`
- 或多数机制「加入时承诺 ≥ 3%」但「实测 < 1%」

**黄灯 🟡**：
- 部分机制 `Gap > 3%`
- 或近期 2 轮性能曲线接近平台

**绿灯 🟢**：
- 所有机制 `Gap < 3%`
- 机制数 vs 性能曲线仍在上升

**判定要点**：
> 研究债务是**无声积累**的。每加一个机制都有"理由"，但累计起来就是功能蠕变。只有持续维护 `DebtTable` 才能看见。

---

### 维度 6：SOTA 架构对比

**采集指标**：
- `N_mech_SOTA`：SOTA 算法的机制数
- `MechRatio` = `N_mech_mine / N_mech_SOTA`
- SOTA 骨架的关键架构特征清单（如 TAEFS 的双档案 + angle split）
- 我方骨架是否具备相同架构特征

**红灯 🔴**：
- `MechRatio ≥ 2` 且主指标仍被 SOTA 压制
- 或诊断显示"SOTA 的关键架构特征我方完全没有，且无法通过加单机制补上"

**黄灯 🟡**：
- `MechRatio ∈ [1.5, 2)`
- 或存在结构性缺口但尚未证伪"加 1 个机制能补"

**绿灯 🟢**：
- `MechRatio ≤ 1.2`
- 或虽然 SOTA 更复杂但我方指标追平或胜出

**判定要点**：
> 当 SOTA 用 3 个机制打赢我方 15 个机制时，这不是"我方调参不够"，是**骨架错了**。在错骨架上加补丁永远救不回来。典型案例：TAEFS 双档案架构 vs 单档案 + 15 机制补丁。

---

### 维度 7：失败边界

**采集指标**：
- `ConsecutiveFailures`：最近连续"同方向"失败的迭代次数
- `FailureRootCause`：每次失败的根因诊断
- `SameRootCauseRun`：失败根因是否指向同一架构限制

**红灯 🔴**（强制止损）：
- `ConsecutiveFailures ≥ 3` 且 `SameRootCauseRun = true`
- 此时**禁止**继续在同一骨架上新增机制，必须转换方向（减法重构 / 换骨架 / 降低 claim）

**黄灯 🟡**：
- `ConsecutiveFailures = 2` 且原因相似
- 继续迭代需先补诊断证据

**绿灯 🟢**：
- `ConsecutiveFailures ≤ 1`
- 或失败根因分散（不同机制的不同问题）

**判定要点**：
> "再试一次"是最昂贵的决策。三次连续同方向失败已经是统计意义上的"不是运气问题"。强制用失败边界机制止损，才能把研究者从盲目碰运气中拉出来。

---

## 三灯综合判定规则

- **任一维度红灯** → 总判定红灯 🔴：停止加机制，进入减法 / 重构 / 止损模式
- **无红灯但 ≥ 2 个黄灯** → 总判定黄灯 🟡：可继续但必须先补齐黄灯对应的证据
- **最多 1 个黄灯** → 总判定绿灯 🟢：可继续正常迭代

**例外覆盖规则**：
- 维度 7（失败边界）红灯具有**一票否决权**，即使其他都绿灯也必须止损
- 维度 3（性能胜出）红灯 + 维度 4（Runtime）红灯同时出现时，必须评估是否放弃当前框架

---

## 硬性禁止行为（触发即自动红灯）

1. **一次迭代新增 ≥ 2 个机制**——违反"最小可验证改动"原则
2. **新机制合并前未跑独立消融**——违反可验证性
3. **红灯状态下继续新增机制**——违反止损原则
4. **机制注册表 / DebtTable 缺失**——无法做研究债务审计
5. **不跑联合消融就声称 ≥ 5 个机制都必要**——无证据堆叠
6. **连续 3 次同方向失败后仍新开同方向分支**——盲目碰运气
7. **runtime 膨胀至 ρ ≥ 3 仍未做复杂度分析**——工程失控
8. **把论文里无法正式定义、无法伪代码化、无法消融对应的机制写成主创新点**——创新 claim 不可读

## 审计工作流（6 步）

1. **确认审计对象**：明确迭代标签、基线、SOTA 对照、数据集范围
2. **数据采集**：按 7 个维度逐一采集指标（优先从已有 CSV / Tracing 抓，缺失就标注"待补"；维度 2 额外检查机制是否可写成论文方法节）
3. **三灯判定**：按阈值打三灯
4. **红灯原因追溯**：每个红灯必须写清根因和证据
5. **下一步建议**：按"立即 / 本周 / 本月"分级
6. **落盘 + 回写 Tracing**：生成 Markdown 报告，同步更新项目 `Tracing/` 或 `skill-outputs/research-iteration-audit-zh/`（根目录 legacy-flat 兼容）

缺失数据处理：
- 若某指标无法采集（如从未做过联合消融），**标记"未采集"并自动降一级判定**（绿 → 黄，黄 → 红）
- 不允许因为"没数据"就跳过维度，必须明确列出"需要补什么实验"

## 通用约束

1. **不编造采集值**——没做过的消融不给数字
2. **宁可苛刻不可宽松**——阈值偏向"发现问题"而非"通过检查"
3. **失败边界 > 其他维度**——一票否决
4. **所有输出中文**
5. **报告必须可被非 AI 读懂**——审稿人 / 导师 / 合作者都能看懂

## 自我迭代模式

### 触发条件

当用户在使用本技能后给出反馈（如"这个阈值太严""维度X不够用""建议加一个XX维度"），或主动说"优化这个 skill"时，进入自我迭代模式。

### 迭代工作流

1. **收集反馈**：哪个维度 / 哪个阈值 / 哪个禁令不合理
2. **诊断根因**：阈值是否太苛刻或太宽松；是否缺少必要维度；是否存在重复判定
3. **提出修改方案**：原文 → 修改后，说明理由
4. **用户确认**：修改方案经用户确认后才执行
5. **写入 SKILL.md**：修改直接应用
