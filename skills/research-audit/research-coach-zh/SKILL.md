# 科研辅助全流程技能

## 技能目标

为"科研问题分析→算法改进→实验→结果分析→进度汇报"的完整科研工作流提供结构化辅导，确保每个环节产出可复现、可追溯的高质量成果。

## Markdown 交付要求

1. 使用本技能完成正式输出时，除聊天中的简要说明外，必须在当前工作区新建一个 Markdown 文档保存完整结果。
2. 默认保存位置为当前工作区 `skill-outputs/research-coach-zh/`（与 `artifact-curator-zh` 按技能分子目录一致；根目录为 **legacy-flat** 兼容）。若该子目录不存在须先创建。
3. 默认文件名格式为 `中文主题_YYYYMMDD_HHMMSS.md`（置于上述目录下；文件名不再重复 skill 名）；若主题不明确，使用 `结果_YYYYMMDD_HHMMSS.md`。
4. Markdown 文档必须包含完整问题分析、算法方案、实验结果分析、对比/消融结论与后续建议。
5. 若用户明确指定保存路径或文件名，以用户要求为准；若用户明确要求只在对话中回答，可跳过创建文件。
6. 最终回复中要说明新建的 Markdown 文件路径（须含 `skill-outputs/research-coach-zh/` 前缀），以及文件里包含的主要内容。

## 模式A：问题约束分析

### 顶刊级约束标准

1. **单痛点原则**：一个研究只解决一个核心痛点，不做瑞士军刀
2. **单方案原则**：一个痛点对应一条主方案线，不堆叠多机制碰运气
3. **可量化原则**：痛点必须有指标可衡量改善程度
4. **可消融原则**：方案的每个子机制必须可独立关闭并验证贡献

### 分析输出结构

```
1. 核心痛点（一句话）
2. 痛点量化指标 + baseline 水平
3. 根因分析（为什么现有方法解决不好）
4. 方案主线（一条）
5. 子机制列表（每个要可独立消融）
6. 预期改善范围
7. 最大风险点
```

### 约束检查清单

- [ ] 痛点是否只有一个？多余的痛点是否已剥离？
- [ ] 方案是否只有一条主线？
- [ ] 每个子机制是否都可独立消融？
- [ ] 改善指标是否可量化？
- [ ] 是否存在循环依赖的子机制？

## 模式B：算法改进编码

### 5步工作流

| 步骤 | 内容 | 产出 |
|------|------|------|
| 1. 现状诊断 | 找到 baseline 代码中的弱点/瓶颈 | 弱点列表 + 证据 |
| 2. 机制设计 | 设计改进机制，每个可独立开关 | 机制描述 + 参数清单 |
| 3. 编码实现 | 写代码，保持与 baseline 同风格 | 可运行代码 |
| 4. 消融验证 | 每个机制单独关闭，验证贡献 | 消融表 |
| 5. 参数敏感度 | 关键超参扫描 | 敏感度曲线/表 |

### 编码铁律

1. **先跑通，再优化**——不要一步到位
2. **保持 baseline 代码风格**——变量名、缩进、数据结构统一
3. **改进代码集中在独立模块**——不散落在 baseline 各处
4. **每个机制有开关变量**——不改一行就能回退到 baseline
5. **不造假数据、不硬编码结果**

### 消融实验规范

- 消融 = 关闭单一机制，其余不变
- 每次消融 HV/指标 drop ≥ 3% 才算有效贡献
- drop < 3% → 先做减法（拆分子通路，识别抵消效应），再做加法
- 消融结果必须含：Full、-M1、-M2、…、baseline
- 表格格式统一，含均值 ± 标准差

## 模式C：结果分析

### 双视角分析框架

**视角1：均值分析（mean）**
- 谁最好 / 最差
- 改进幅度（绝对值 + 百分比）
- 排名稳定性（跨问题 / 跨数据集）

**视角2：稳定性分析（std / IQR）**
- 方差大小
- run 间一致性
- 是否存在个别 run 异常拉低/拉高

### 分析输出结构

```
1. 一句话结论
2. 均值对比表（含 rank）
3. 稳定性对比表
4. 显著性/置信度评估（如有统计检验）
5. 逐问题/逐数据集拆解
6. 异常点标注与解释
7. 改进建议（下一步做什么）
```

### 对比实验报告约束

- 不挑数据：所有问题/数据集都要报，不能只报好的
- 不模糊：不用"略好""差不多"，必须给具体数值
- 不隐瞒弱项：弱项明确指出 + 分析原因
- 表格必须含均值 ± 标准差

## 模式D：周报/日报生成

### 模板结构（严格遵循，不增不改）

```markdown
# 本周工作总结（YYYY.MM.DD - YYYY.MM.DD）

## 一、本周完成工作
### 1.1 [工作项1名称]
- 具体内容（带数据/版本号/文件名）
- 成果（指标变化/产出物）

### 1.2 [工作项2名称]
...

## 二、遇到的问题与解决
| 问题 | 原因分析 | 解决方案 | 状态 |
|------|----------|----------|------|

## 三、下周计划
| 序号 | 计划内容 | 预期产出 | 优先级 |
|------|----------|----------|--------|
```

### 周报规则

1. 数据必须来自实际操作，不编造
2. "完成工作"部分只写做了什么 + 具体结果，不写过程流水账
3. 问题与解决必须填，没有则写"本周无阻塞问题"
4. 下周计划必须可验证（有预期产出）
5. 全文中文

## 通用约束

1. **不编造实验数据**——没跑过的实验不出结论
2. **不跳步**——按模式内步骤顺序执行
3. **信息不足时先列出缺什么**，提示用户补充
4. **所有分析输出中文**
5. 英文术语后附中文翻译，如 `ablation（消融）`

## 自我迭代模式

### 触发条件

当用户在使用本技能后给出反馈（如"这部分不好用""输出格式需要改""多了/少了XX"），或主动说"优化这个skill"时，进入自我迭代模式。

### 迭代工作流

1. **收集反馈**：明确用户不满意的具体环节（哪个模式 / 哪个步骤 / 什么问题）
2. **诊断根因**：是规则缺失、规则冲突、粒度不够、还是场景未覆盖？
3. **提出修改方案**：列出拟修改的条目（原文 → 修改后），说明修改理由
4. **用户确认**：修改方案经用户确认后才执行
5. **写入 SKILL.md**：将修改直接应用到本技能文件
6. **记录迭代日志**：在 `references/iteration-log.md` 追加本次迭代记录

### 迭代日志格式

```markdown
## [日期] 迭代 #N

- **触发反馈**：...
- **根因**：...
- **修改内容**：...
- **影响范围**：模式X / 步骤Y
```

### 迭代约束

- 不破坏现有模式结构——增量修改，不推倒重写
- 每次迭代只改一个关注点——不趁机大改
- 修改后 SKILL.md 总行数仍 < 500
- 重大结构变更（如新增/删除模式）必须用户明确同意
